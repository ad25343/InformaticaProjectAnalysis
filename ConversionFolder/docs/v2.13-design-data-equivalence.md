# Design Document: v2.13 — Data-Level Equivalence Testing

**Status:** Draft
**Author:** Conversion Tooling Team
**Date:** 2026-03-04
**Target version:** 2.13.0

---

## 1. Problem

The tool currently validates converted code through four structural layers:

| Layer | What it checks | What it misses |
|---|---|---|
| Smoke execution (Step 10a) | Syntax, YAML validity, SQL structure | Logic correctness |
| Structural reconciliation (Step 10b) | Target fields, source tables, expressions present | Whether values match |
| Coverage / verification flags (Step 4, 9) | Field coverage %, lineage gaps | Runtime behaviour |
| Generated tests (Step 9) | Column presence, null rates, non-empty output | Row-level value equivalence |

None of these answer the key question: **for a given input row, does the generated code produce the same output value as the original Informatica session?**

This gap is most acute in four expression categories identified as high-risk:

1. **IIF / DECODE** — NULL-branch semantics and fallthrough differ between Informatica and Python/SQL
2. **Date arithmetic** — `ADD_TO_DATE`, `DATE_DIFF`, `TO_DATE` format strings vary by target stack
3. **String functions** — `SUBSTR` is 1-indexed in Informatica, 0-indexed in Python; `INSTR` returns 0 vs -1 for not-found
4. **Aggregations / window functions** — partition key ordering and NULL treatment differ across Spark, dbt, and SQL

---

## 2. Proposed Solution

Two additions to Step 9's test output, shipping with every converted job:

### Component A — Expression Boundary Tests

Auto-generated pytest test functions that test each high-risk transformation expression **in isolation** against a small set of boundary inputs.

- No external data required — boundary values are synthetic
- Works at conversion time, no database connection needed
- Designed to catch the most common mistranslation patterns before UAT

### Component B — Golden CSV Comparison Script (`compare_golden.py`)

A self-contained Python script generated for every job. The team runs it locally after capturing Informatica's output and running the generated code against the same inputs.

- Compares field-by-field: match rate, null rate, value distribution
- Reports mismatched rows (sample of up to 20)
- Produces pass/fail against a configurable tolerance threshold (default: 100%)
- No tool changes required to run — plain `python compare_golden.py --expected X --actual Y`

## 3. Component A — Expression Boundary Tests

### 3.1 Detection

`test_agent.py` already parses `conversion_output.files` to discover models and tables. In v2.13, it will additionally scan `graph["mappings"][*]["transformations"][*]["expressions"]` to identify high-risk expressions by pattern matching.

Detection patterns (applied to `EXPRESSION` field strings):

| Risk category | Detection pattern | Example |
|---|---|---|
| IIF | `IIF\s*\(` | `IIF(AMOUNT > 0, AMOUNT, 0)` |
| DECODE | `DECODE\s*\(` | `DECODE(STATUS, 'A', 'Active', 'Unknown')` |
| Date function | `ADD_TO_DATE\|DATE_DIFF\|TO_DATE\|TRUNC` | `ADD_TO_DATE(LOAD_DATE, 'MM', 1)` |
| String function | `SUBSTR\|INSTR\|LTRIM\|RTRIM\|LPAD\|RPAD` | `SUBSTR(CODE, 1, 3)` |
| Aggregation | `SUM\s*(\|AVG\s*(\|COUNT\s*(\|MAX\s*(\|MIN\s*(` | `SUM(AMOUNT)` |

### 3.2 Test Cases Generated Per Category

#### IIF
```python
# Generated for: IIF(AMOUNT > 0, AMOUNT, 0)
@pytest.mark.parametrize("amount,expected", [
    (100.0,  100.0),   # TRUE branch — normal value
    (-5.0,   0.0),     # FALSE branch
    (0.0,    0.0),     # boundary — exactly zero
    (None,   0.0),     # NULL input — Informatica evaluates condition as FALSE → FALSE branch
])
def test_expr_iif_AMOUNT_null_handling(amount, expected):
    """IIF NULL handling: Informatica treats NULL condition as FALSE, not as NULL propagation."""
    result = iif_amount(amount)   # calls the translated function
    assert result == expected, f"IIF mismatch: input={amount}, got={result}, expected={expected}"
```

Key assertion: NULL input to IIF condition evaluates the FALSE branch in Informatica — this differs from SQL `CASE WHEN` (which propagates NULL) and must be explicitly tested.

#### DECODE
```python
# Generated for: DECODE(STATUS, 'A', 'Active', 'I', 'Inactive', 'Unknown')
@pytest.mark.parametrize("status,expected", [
    ('A',    'Active'),     # first match
    ('I',    'Inactive'),   # second match
    ('X',    'Unknown'),    # default
    (None,   'Unknown'),    # NULL — Informatica DECODE treats NULL as no-match → default
    ('a',    'Unknown'),    # case-sensitive — Informatica DECODE is case-sensitive
])
def test_expr_decode_STATUS(status, expected):
    """DECODE: case-sensitive matching; NULL falls through to default."""
    ...
```

#### Date arithmetic
```python
# Generated for: ADD_TO_DATE(LOAD_DATE, 'MM', 1)
@pytest.mark.parametrize("load_date,expected", [
    (date(2024, 1, 31), date(2024, 2, 29)),  # month-end rollover (leap year)
    (date(2024, 1, 15), date(2024, 2, 15)),  # normal case
    (None,              None),               # NULL propagation — date funcs return NULL on NULL input
])
def test_expr_add_to_date_LOAD_DATE(load_date, expected):
    """ADD_TO_DATE: verify month rollover and NULL propagation."""
    ...
```

#### String functions
```python
# Generated for: SUBSTR(PRODUCT_CODE, 1, 3)
@pytest.mark.parametrize("code,expected", [
    ('ABCDEF', 'ABC'),   # normal — 1-indexed start in Informatica
    ('AB',     'AB'),    # shorter than length — Informatica returns available chars
    ('',       ''),      # empty string
    (None,     None),    # NULL propagation
])
def test_expr_substr_PRODUCT_CODE(code, expected):
    """SUBSTR: Informatica is 1-indexed; Python is 0-indexed — verify offset correction."""
    ...
```

#### Aggregations
```python
# Generated for SUM(AMOUNT) with GROUP BY ACCOUNT_ID
def test_expr_sum_AMOUNT_empty_partition(spark):
    """Aggregation over empty partition must return NULL (not 0) — matches Informatica."""
    df = spark.createDataFrame([], schema)
    result = run_pipeline(df)
    assert result.count() == 0 or result.filter(col("SUM_AMOUNT").isNull()).count() > 0
```

### 3.3 Output

Tests are appended to the existing `tests/test_{mapping_name}.py` file (PySpark/Python jobs) or added as a separate `tests/test_expressions_{mapping_name}.py` for dbt jobs (where the main test file uses SQL).

## 4. Component B — Golden CSV Comparison Script

### 4.1 Purpose

The team captures a sample output from the live Informatica session (any number of rows, typically 100–10,000 for spot-checking), runs the generated code against the same source rows, and runs `compare_golden.py` to diff the results.

### 4.2 Usage

```bash
python tests/compare_golden.py \
  --expected informatica_output.csv \
  --actual generated_code_output.csv \
  [--threshold 99.5] \
  [--key-columns ACCOUNT_ID,LOAD_DATE] \
  [--ignore-columns AUDIT_TIMESTAMP,ETL_RUN_ID]
```

| Argument | Default | Description |
|---|---|---|
| `--expected` | required | CSV captured from Informatica session |
| `--actual` | required | CSV from running the generated code |
| `--threshold` | `100.0` | Minimum % of rows that must match exactly to pass |
| `--key-columns` | auto-detect | Columns used for row-level join before comparison |
| `--ignore-columns` | none | Columns to exclude (e.g. audit timestamps that will always differ) |

### 4.4 Generated Script Structure

The script is entirely self-contained (stdlib + pandas only). It is generated once per job and saved as `tests/compare_golden.py`. It does not require the conversion tool to run — it runs anywhere pandas is available.

Key sections:
1. Argument parsing
2. Load both CSVs into pandas DataFrames
3. Schema comparison (column presence, order)
4. Row count comparison
5. Key-column-based join (or positional join if no key specified)
6. Field-by-field value comparison with tolerance for floats (`rtol=1e-5`)
7. Null rate comparison per column
8. Mismatch sample (top 20 rows sorted by key columns)
9. Pass/fail against threshold
10. Likely-cause heuristics for common mismatches (float rounding, date format, off-by-one)

### 4.5 Likely-Cause Heuristics

The script includes a small heuristic engine that tags common mismatch patterns:

| Pattern | Heuristic applied |
|---|---|
| Numeric columns with mismatches < 0.01 absolute diff | "Float rounding — review IIF/arithmetic expressions" |
| Date columns with mismatches | "Date format mismatch — check TO_DATE format string vs target stack" |
| String columns with mismatches on whitespace trim | "LTRIM/RTRIM — Python strips both ends by default, Informatica strips one" |
| Row count mismatch (actual > expected) | "Possible fan-out — check JOIN conditions for inadvertent cartesian product" |
| Row count mismatch (actual < expected) | "Possible data loss — check filter conditions and NULL handling" |

## 6. Limitations

**Component A (expression tests) will not catch:**
- Logic errors in expressions that don't involve the specific boundary values tested
- Differences caused by source data distributions (e.g. edge cases in production data not covered by boundary inputs)
- Stack-specific behaviour differences not captured by the 4 risk categories

**Component B (golden comparison) requires:**
- The team to have a live Informatica environment to capture golden output
- Sample data representative of real production inputs
- The generated code to be runnable locally or in a staging environment

Neither component replaces full UAT — they complement it by making the most common failure modes visible before production deployment.

---

## 7. Future Work (out of scope for v2.13)

- **Inline execution**: run the generated code automatically against synthetic data in a sandboxed environment and compare outputs without requiring user action (blocked on environment isolation complexity)
- **Automated golden capture**: API integration with Informatica Repository Manager to trigger a test session run and capture output programmatically
- **Tolerance profiles**: per-column tolerance configuration (e.g. allow ±0.01 on financial columns, exact match on key columns)
- **Regression tracking**: store comparison results in the job database and alert when a re-conversion changes the match rate
